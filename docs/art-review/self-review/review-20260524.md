**Manuscript:** Architecting the Digital RES Learning Factory: A Scalable MING+React Telemetry Pipeline for Multi-Vector Energy Systems
**Review date:** 24 May 2026
**Spec version:** docs/pl-Edu-Bad-specification/IoT_Data_Arch-JSON-MQTT.md (current working copy)

## Major Revisions

> Issues that would cause outright rejection or a mandatory full re-review cycle if unaddressed.

M1. [§4.2 JSON Payload Schema] — **spec divergence**
"The schema is anchored by mandatory root keys: `device_id`, `device_type`, `timestamp`, and a `data` object" — The manuscript's Canonical Data Model — the paper's declared "core architectural contribution" — does not match the canonical envelope in the specification. Every category CDM block and the Physical Device Onboarding Contract in the spec define the required top-level keys as `measurement`, `tags`, `fields`, `timestamp`, `status`, with metadata nested inside a `tags` object and telemetry inside a `fields` object, and timestamps as Unix epoch milliseconds. The manuscript instead uses `device_id`/`device_type`/`data` root keys, places metadata at the root ("These example payloads do not nest metadata keys under a parent `tags` object"), and uses ISO-8601 string timestamps. The central contribution is therefore described with a schema that contradicts the source of truth. Required fix: either re-derive §4 from the spec's actual envelope, or, if the manuscript schema is a deliberate redesign, state that explicitly and reconcile it back into the spec — silent divergence on the headline artefact is not acceptable.

M2. [§4.3 Engine Bench Payload Example] — **spec divergence**
"`grid_voltage_v`: 230.1, `grid_current_a`: 6.8, `grid_frequency_hz`: 50.01" — The engine-bench worked example assigns `active_power_kw`, `grid_voltage_v`, `grid_current_a`, and `grid_frequency_hz` to the engine device. None of these appear in the spec's Category 6 (Engine Bench) CDM, whose electrical/mechanical fields are `engine_speed_rpm`, `torque_nm`, and `measured_power_kw`; the grid voltage/current/frequency fields belong to the spec's Category 1 (Photovoltaic) CDM. The `device_id` value `engine_bench_1` also does not match the spec's engine station identifier (`bench_r121`). The worked example is a composite that does not correspond to any single spec category. Required fix: rebuild the example strictly from the Category 6 CDM, or justify the deviation against the spec.

M3. [§4.4 Algae Photobioreactor Payload Example] — **spec divergence**
"`gas_pressure_kpa`: 105.2, `methane_ch4_percent`: 0.05" — The algae photobioreactor worked example contains `gas_pressure_kpa` and `methane_ch4_percent`, which are anaerobic-digestion parameters from the spec's Category 3 (Biogas) CDM; the spec's Category 5 (Algae) CDM contains neither (algae reactors are monitored for dissolved O2/CO2, not methane). The field `medium_temperature_c` is also a renaming of the spec field `temperature_medium_c`, and `culture_strain` is given as `chlorella` where the spec uses `chlorella_vulgaris`. Required fix: rebuild the example from the Category 5 CDM; a methane field in an algae reactor will be read by a domain reviewer as a technical error.

M4. [§3.3 MQTT Topic Taxonomy and Security Model] — **spec divergence**
"A multi-level hierarchy is implemented that follows a strict spatial and logical progression: `wksir/kioze/{category}/{device_id}/{stream}`" — The manuscript states the canonical taxonomy "is implemented" and that the ingestion gateway "is orchestrated to utilize" wildcard subscriptions `wksir/kioze/+/+/telemetry`. The spec's Gap 1 (flagged *Critical*) states the opposite: the implemented Node-RED flows and ACL use the prefix `devices/<category>/<device_id>/telemetry`, and "A physical device publishing to `wksir/kioze/...` hits no subscriber … Data is silently dropped." The manuscript presents an unimplemented target design as operational. (There is also an internal inconsistency: the taxonomy lists streams `telemetry, status, alarms, cmd`, while the ACL prose in the same section uses `data`, `status`, `commands`.) Required fix: describe the `devices/` prefix as actually implemented and present `wksir/kioze/` as the migration target, consistent with Gap 1.

M5. [§4.2 JSON Payload Schema / §3.2 Decide] — **maturity over-claim**
"this phase performs a structural audit on every MQTT packet—if validation fails, the data is dropped" — The manuscript repeatedly claims the pipeline validates and rejects malformed telemetry ("real-time structural audits", "structural audit on every MQTT packet"). The spec's Gap 3 states plainly that input validation/schema enforcement is *missing*: "The CDM transformer trusts whatever JSON arrives. A physical device sending malformed payloads, out-of-range values, or extra fields passes straight to InfluxDB without rejection or alerting." The manuscript also attributes this to a named flow, `v3.0-cdm-ingestion-gateway.json`, which the spec does not contain — and Gap 2 states there is no dedicated ingestion flow at all. Null-key pruning (deleting `null` fields) is real per the spec, but it is not packet-level schema validation; the manuscript conflates the two. Required fix: separate the operational null-pruning from the absent schema-validation layer, and disclose Gap 3.

M6. [§5.3 Performance Results / Table 2 / Abstract] — **unverified benchmark**
"the CDM null-pruning transformer successfully maintained a 100% data completeness rate for all valid telemetry packets" — None of the headline metrics — 100% valid-packet integrity, 100% null-pruning accuracy, 42.5 ms average API latency, 99.8% end-to-end success, and the "less than 1 ms" transformation overhead — appear in the specification, and no measurement artefact, log, or instrumentation dataset accompanies the submission. The spec is an architecture/schema document containing no benchmarks and explicitly describing a "simulation MVP". The numbers are nonetheless presented as "Experimental results confirm …" with a results table carrying a PASSED/OPTIMAL status column. Per Rule P2, every quantitative claim must trace to a reference, the spec, a configuration file, or a recorded measurement; these trace to none. The project's own writing plan (§5.5) explicitly directed that this section be framed as "simulation-based architectural validation, not a hardware benchmark" — the manuscript does the opposite. Required fix: either supply the measurement methodology and data, or remove the quantitative results entirely.

M7. [§5.3 / §6 / §7 — maturity language] — **maturity over-claim**
"the results proved the architecture is production-grade" — The manuscript describes the system as "production-grade" (§5.3), "a proven, production-grade telemetry solution" (§7), and the abstract states "These findings prove the architecture's suitability". The spec states: "The current deployment is a **simulation MVP**." A loopback-only containerised simulation with no physical-device integration, no schema-validation node, and an unaligned MQTT taxonomy cannot be called production-grade or "proven". Required fix: replace all production-grade/proven/"prove" language with accurate maturity statements (simulation MVP, architectural prototype).

M8. [§5.3 Performance Results / §3.2 Decide] — **spec divergence**
"zero packet loss captured across the twelve local simulation flows" — The manuscript states there are twelve simulation flows, and §3.2 claims "schema consistency across all 12 measurements." The spec states Node-RED runs **10 flows** and the InfluxDB Schema Reference lists **10 measurements**. The manuscript conflates its own "12 vectors" abstraction (the 12 parameters in Table 1) with the flow and measurement counts. Required fix: correct the counts to 10, and clarify that the "12 vectors" is a curated parameter subset, not the platform's flow/measurement inventory.

M9. [§3.3 Security Model] — **spec divergence**
"The broker is bound exclusively to the internal Docker bridge network" — The manuscript claims the MQTT broker "is not exposed to the public internet" and is bound only to the internal bridge network. The spec's Container Summary maps `iot-mosquitto` to external host port 40098 (production) and 1883 (local development). A host port mapping contradicts the claim of exclusive internal binding. Required fix: describe the broker's actual port exposure per the spec, or correct the spec if the manuscript is right — the two must agree.

M10. [Abstract / §5 / §7 — journal fit] — **journal-fit failure**
"These findings prove the architecture's suitability as a highly resilient, cross-domain research environment" — The contribution is an educational, simulation-only telemetry pipeline for a renewable-energy learning factory. That scope is disclosed honestly in the Limitations passages, but it is concealed in the high-visibility parts of the paper — the abstract, the Performance Results section, and the Conclusions all read as a validated, production-grade deployment. A Section A1 referee judging "Smart Grids and Microgrids" relevance needs the educational-simulation scope stated plainly in the abstract and the results framed as architectural rather than empirical. The keyword list also omits "smart grid" and "microgrid", which weakens the section placement. Required fix: state the simulation-MVP scope in the abstract; align the results and conclusions framing with the limitations sections.

M11. [§6.3 Comparison to Related Approaches] — **load-bearing citation failure**
"the hydroponics-focused MING stack described in earlier studies [@budiman2023study]" — The entire Comparison section is anchored on `@budiman2023study` as "the baseline system", against which the null-pruning logic and corridor algorithm are said to be benchmarked and to show "high transmission reliability." The cited work is *Study of the Effect of Physical Parameters on Commercial Hydroponics Based on IoT: A Case Study of Bok Coy Plants and Water Spinach* — a horticultural study, not a MING-stack telemetry-pipeline paper, and not a plausible source for a transmission-reliability benchmark. The bib entry also lacks a DOI and volume (only a SciSpace URL). A central comparative claim resting on a citation that does not support it is a load-bearing failure. Required fix: substitute a genuine MING/telemetry-pipeline baseline, or remove the comparative claims.

M12. [Front matter — Funding / Data Availability] — **submission blocker**
"`fundingstatement: | [...]`" and "`dataavailability: | [...]`" — Both the Funding statement and the Data Availability statement are unfilled `[...]` placeholders in the YAML front matter. MDPI requires a completed Funding statement (and Data Availability statement) at submission. Required fix: complete both statements before submission. (The Conflicts of Interest statement and the AI-use disclosure in Acknowledgments are present and adequate — see the Minor section.)

## Minor Revisions

> Issues that must be fixed before acceptance but do not require a full re-review.

m1. [Abstract / §3 / §4 — acronyms]
"an optimal average API response latency of 42.5 ms" — Several acronyms are not expanded on first use: **API** (first appears in the abstract), **JSON**, **MQTT**, **UI**, **PV**, **WAN**, **M2M**. Conversely, **RES** and **SCADA** are each expanded twice (abstract and Introduction), and **RES** is expanded inconsistently — "renewable energy systems" in the abstract versus "Renewable Energy Source" in the Introduction. Fix: expand each acronym once, at first use, consistently.

m2. [Multiple — sentence length]
"While similar virtualized digital twin environments report latencies exceeding 100 ms [@caiza2023digitaltwin] due to the heavy synchronization overhead …" — The five worst over-length sentences (>35 words): (1) §5.3, the "While similar virtualized digital twin environments …" sentence (~44 words); (2) §7 Summary of Achievements, "By achieving 100% null-pruning data integrity and recording an average API latency …" (~46 words); (3) §5.3, "Finally, system resiliency was evaluated, verifying an end-to-end success rate of 99.8% …" (~37 words); (4) §6.2, "Because the middleware dynamically maps root-level metadata keys … to InfluxDB tags …" (~36 words); (5) §3.3, "This hierarchical approach enables a modular expansion capability where adding a new energy asset …" (~36 words). Fix: split each into two sentences.

m3. [Back matter — References]
"bibliography.bib contains 24 entries; all 24 are cited" — The reference list holds 24 entries, below the ≥25 typically expected of an MDPI full research article and far below this project's own writing-plan target of 50–80. (Reciprocal check passed: all 24 `@key` citations resolve to `bibliography.bib`, and there are no orphan entries.) Fix: expand the reference base, particularly in Related Work.

m4. [Figure 3 caption]
"The plot demonstrates how the simulation preserves physical inertia (Temporal Continuity)" — The Figure 3 caption does not define the plot axes or their units, does not state the time window, and does not identify which of the 12 vectors is plotted. "Temporal Continuity" and "Stochastic Volatility" appear to be legend labels but are not introduced. Fix: make the caption self-contained — axes, units, vector shown, window. (Figures/tables checks that passed: Figures 1–3 are each cross-referenced in the body before they appear; Tables 1–2 are numbered, titled, and carry units in their columns/values.)

m5. [Table 1 vs §5.1 — internal contradiction]
"published these heartbeats to the MQTT broker at three-second intervals" — §5.1 states a three-second publish interval, while Table 1 gives ingest frequencies of 1000 ms (fast vectors) and 5000 ms (slow vectors), and the spec specifies an inject timer "every 3–5 s". Three figures, all different. Fix: reconcile the interval across Table 1, §5.1, and the spec.

m6. [§6.1 Scalability — internal contradiction]
"The InfluxDB 2.x storage engine was benchmarked to handle increased sensor density without performance degradation" — This sentence asserts a completed scalability benchmark; the sentence two lines later concedes "future benchmarking is required to evaluate scalability bottlenecks." Fix: remove the benchmark claim or substantiate it.

m7. [End of manuscript — Scratch section]
"# Scratch {#sec-scratch}" — An empty `# Scratch` heading remains at the end of `article.qmd`. With `number-sections: true`, it will render as a numbered section after Conclusions. Fix: delete the Scratch heading and its placeholder comment before submission.

m8. [Front matter — authorship]
"The author declares no conflict of interest" — The manuscript lists a single author, while `CLAUDE.md` and the writing plan name Viktar Taustyha as senior author alongside the co-author, and the writing plan §9 gives a two-author CRediT division. The Conflicts of Interest statement is singular ("The author"), while the Acknowledgments is plural ("The authors used Claude"). Fix: confirm the intended author list and make the author block, CRediT roles, and back-matter voice consistent.

m9. [Front matter — ORCID]
"name: Kanchipogu Kelvyn G Melcheizedek … corresponding: true" — No ORCID is provided for the corresponding author. MDPI requires an ORCID for the corresponding author. Fix: add the ORCID iD(s).

m10. [Abstract / §1 / §4 — register]
"functions as a \"Universal Translator\"" — The manuscript leans heavily on coined metaphors ("Impedance Mismatch", "Universal Translator", "Nervous System", "heartbeats", "pulse"). For an A1 research article this register is informal, and "Impedance Mismatch" is presented in the abstract as an established condition ("the Impedance Mismatch inherent in multi-energy microgrids") when it is the authors' own coinage, defined only later in §1. Fix: reduce metaphor density; on first use, mark coined terms as the authors' own.

m11. [§1 Introduction / §2 Related Work]
"Standard approaches often omit real-time structural audits and data cleansing layers" — The Introduction carries only three citations and states no sharp, isolated research-gap sentence; it also omits the explicit C1/C2/C3 contribution statement the writing plan §5.1 specified. The Related Work subsections are short (1–2 citations each), and the most on-topic prior works (`@dani2025aipowered`, `@silva2022development`, `@kychkin2019architecture`) are cited in passing in later sections rather than engaged in Related Work. Fix: add an explicit gap statement and enumerated contributions; deepen Related Work. (Formatting checks that passed: cross-references use Quarto `@fig-`/`@tbl-` syntax with no bare numbers; no first-person singular; no `<!-- @TODO-cite -->` or `<!-- gap: -->` markers remain in the body; the AI-use disclosure in Acknowledgments is present and covers drafting, structural editing, figure source generation, bibliography auditing, and human verification; IMRaD section order is otherwise compliant.)

## Methodological Gaps

> Missing detail that prevents reproducibility or undermines the novelty claim, even if the prose is otherwise correct.

G1. [Corridor algorithm — §5.1] — The "Strict Corridor Algorithm" is described only as "a recursive random walk where each data point is constrained to a ±2% window relative to its predecessor." There is no equation, no pseudocode, and no flow diagram. The ±2% threshold is asserted, not derived or justified; per-vector seed/initialisation values, the random distribution used, and failure modes are unstated. The spec describes the platform simulation as "physics-based mathematical models", not a bounded random walk — a divergence in itself. A reader cannot reproduce the algorithm. Minimum acceptable: a pseudocode block or recurrence equation, the threshold justification, and initialisation values. (Cross-reference: M6 — the headline results depend entirely on this simulation.)

G2. [Measurement methodology — §5.3] — For the 42.5 ms latency and 99.8% end-to-end success figures, no instrumentation method, sample size, number of trials, or statistical treatment is given (only a single mean per metric; no variance, percentiles, or confidence interval). The "less than 1 ms" overhead has a described method (timestamp hooks) but no data. Minimum acceptable: instrumentation description, N, trial count, and a dispersion measure for each reported value. (Cross-reference: M6.)

G3. [Load conditions — §5.1] — The metrics are reported "under peak load" and "high-concurrency load", but "peak load" is never quantified: messages per second, number of concurrent publishers, and total test duration / packet count are all absent. Minimum acceptable: a defined load profile (rate, concurrency, duration) so the conditions can be replicated.

G4. [Figure 3 grounding — §5.1, Fig. 3] — Figure 3 (corridor algorithm vs. white noise) illustrates the ±2% corridor, which does not appear in `IoT_Data_Arch-JSON-MQTT.md`; the spec specifies physics-based generators. The figure's content cannot be traced to the spec. Minimum acceptable: ground the figure in a defined, documented algorithm (see G1) and state the data source.

G5. [Null-pruning evaluation — §5.3] — The manuscript reports "100% Null-Pruning accuracy for malformed data" but never defines what constitutes a malformed packet, how many were injected, or how detection was scored. The spec's only fault mechanism is a 0.1%-probability fault-injection overlay; the manuscript's evaluation procedure against it is not described. Minimum acceptable: an operational definition of a malformed packet, the injected count, and the scoring rule.

G6. [Novelty isolation — §1, §2, §4] — The CDM, null-pruning, and tag/field-separation design is described in the spec's "Architectural Principles" as the *existing* KIOZE Edu-Bad platform architecture, yet the manuscript presents it as "a novel contribution of this work." The paper does not delineate what is genuinely new versus the cited MING/Node-RED/InfluxDB monitoring systems (`@dani2025aipowered`, `@kychkin2019architecture`, `@silva2022development`). Minimum acceptable: an explicit, falsifiable statement of novelty that separates the paper's contribution from both the pre-existing platform spec and prior art. (Cross-reference: M10.)

## Editor recommendation

*"As a referee for MDPI Energies Section A1, my recommendation is Reject. The primary obstacle to acceptance is that the manuscript's headline experimental results — 100% data integrity, 100% null-pruning accuracy, 42.5 ms average API latency, 99.8% end-to-end success — cannot be traced to any measurement, configuration artefact, or specification supplied with the submission, yet they are presented in the abstract, a results table with a PASSED/OPTIMAL column, and the conclusions as confirmed benchmarks of a 'production-grade' system. Compounding this, the paper's core technical descriptions contradict its own canonical specification: the Canonical Data Model schema, the MQTT topic taxonomy, the input-validation behaviour, the broker network exposure, and the flow/measurement counts are all described as operational when the specification marks them as either a different design or an unimplemented target (the system is, in the spec's own words, a 'simulation MVP', with Gaps 1–3 still open). A manuscript cannot be revised into acceptability when its empirical claims are unverifiable and its system description does not match the system; the necessary fixes are new instrumented measurement work and an honest reframing as a simulation-based architectural proposal, not editorial revision. The underlying KIOZE Edu-Bad platform and the CDM concept are a legitimate basis for a paper — but that paper must report real, instrumented results, describe the system as actually implemented (including the `devices/` topic prefix and the absent schema-validation node), drop the 'production-grade'/'proven' language, and state the educational-simulation scope plainly in the abstract. I would encourage the authors to resubmit on that basis as a new submission."*
